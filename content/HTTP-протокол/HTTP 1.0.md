> [!abstract] **Главное** 
> Переломная версия, превратившая интернет из текстовой среды в мультимедийное пространство. Главное новшество — внедрение **метаданных (заголовков)** и **кодов состояния**, что позволило браузерам передавать информацию о себе, а серверам — сообщать о результатах запроса (например, 200 OK или 404). 
> Протокол перестал ограничиваться только HTML и научился передавать изображения, видео и аудио, фактически став системой передачи гипермедиа. 
> Однако он оставался медленным из-за необходимости открывать новое соединение для каждого отдельного элемента страницы. ^679c15

## HTTP/1.0 поддерживает различный тип контента

HTTP/1.0 разрушил монополию HTML, введя поддержку различных типов контента и кодировок:
- Появилась возможность передавать изображения, видео, аудио и любые другие бинарные файлы.
- Поддержка различных кодировок и многокомпонентных (multi-part) типов данных сделала протокол универсальным. Из-за этого само название «Hypertext» стало технически неточным, так как система переросла рамки простого текста.

```
/* Для HTML-страницы */
Content-Type: text/html; charset=utf-8

/* Для изображения */
Content-Type: image/jpeg

/* Для аудио-файла */
Content-Type: audio/mpeg
```
## В HTTP/1.0 внедрены метаданные и команды управления

Структура сообщения радикально изменилась за счет появления механизмов обмена служебной информацией:
- **HTTP-заголовки:** Впервые позволили клиенту сообщать серверу сведения о браузере, авторизации и ожидаемом типе контента. Это открыло путь к кэшированию и защищенным разделам сайта.
- **Статус-коды:** Трехзначные коды с пояснениями позволили пользователю понимать причину ошибки или успех операции.
- **Новые методы:** К базовому `GET` добавились `POST` (отправка данных из форм) и `HEAD` (запрос только заголовков для проверки метаданных).

```
/* Запрос клиента с метаданными */
GET /image.webp HTTP/1.0
User-Agent: Mozilla/1.0
Accept: image/webp, */*

/* Ответ сервера с типом контента */
HTTP/1.0 200 OK
Content-Type: image/webp
Content-Length: 45023

[Бинарные данные изображения]
```
## HTTP/1.0 сохранил подход «один запрос — одно соединение»
Несмотря на функциональный рывок, производительность оставалась низкой из-за архитектуры «один запрос — одно соединение»:
- Для каждого из 10 картинок на странице браузеру приходилось заново проходить процедуру «трехэтапного рукопожатия» TCP.
- Поскольку протокол работал без сохранения состояния (stateless), в каждом новом соединении клиенту приходилось дублировать одни и те же заголовки, перегружая сеть. Попытки исправить это заголовком `Connection: keep-alive` не были стандартизированы и работали нестабильно.
