**HTTP-запрос (Request)** — это стандартизированный способ, с помощью которого клиентские платформы (например, веб-браузеры) запрашивают информацию, необходимую для загрузки веб-ресурсов. Это сообщение является частью протокола прикладного уровня, работающего поверх стека TCP/IP.

### **Основные компоненты HTTP-запроса**

Типичный запрос включает в себя пять ключевых элементов:

1. **Метод HTTP (HTTP Method):** указывает на действие, которое сервер должен выполнить.
2. **URL (Uniform Resource Locator):** адрес запрашиваемого ресурса.
3. **Версия HTTP:** тип используемого протокола (например, HTTP/1.1, HTTP/2 или HTTP/3).
4. **Заголовки запроса (Request Headers):** метаданные в формате «ключ-значение».
5. **Тело запроса (Body):** необязательный блок с данными, передаваемыми на сервер.

### **Эволюция структуры запроса**

В источниках подробно описывается, как запрос усложнялся с развитием интернета:

- **HTTP/0.9 (1991):** максимально простой «однострочный» протокол. Запрос состоял только из метода **GET** и пути к файлу (например, `GET /index.html`). В этой версии **не было заголовков** и возможности отправлять данные от клиента.
- **HTTP/1.0 (1996):** появились заголовки, что позволило клиенту передавать информацию о себе (тип браузера, ожидаемый формат данных). Были добавлены методы **POST** и **HEAD**.
- **HTTP/1.1 (1997):** расширил список методов (**PUT, DELETE, OPTIONS, PATCH**) и ввел обязательный заголовок **Host**, необходимый для идентификации хоста.
- **HTTP/2 (2015):** кардинально изменил формат запроса, сделав его **бинарным**, а не текстовым. Сообщения стали состоять из **фреймов** (HEADERS для метаданных и DATA для полезной нагрузки). Для оптимизации были введены **псевдо-заголовки**, такие как `:method`, `:scheme`, `:host` и `:path`.
- **HTTP/3 (2022):** сохраняет логику фреймов и потоков, но переходит на транспортный протокол **QUIC** (базирующийся на UDP) для ускорения установления связи.
Развитие версий протокола HTTP коренным образом меняло структуру, возможности и эффективность сетевого запроса. Каждая новая итерация решала проблемы предыдущей, превращая простой обмен текстом в сложную систему бинарных потоков.

##### **HTTP/0.9 (1991): Однострочный запрос**
В самой первой версии протокол был предельно прост и предназначался только для передачи HTML-данных. 
*   **Минимализм:** Запрос состоял из одной строки. Единственным доступным методом был **GET**.
*   **Отсутствие метаданных:** В запросе не было **заголовков**. Это означало, что клиент не мог передать серверу информацию о себе (тип браузера), о желаемой кодировке или куки.
*   **Структура:** Вы просто запрашивали путь к файлу (например, `GET /index.html`), и после передачи контента соединение немедленно закрывалось.

##### **HTTP/1.0 (1996): Появление метаданных**
Эта версия значительно расширила возможности взаимодействия, сделав HTTP универсальным инструментом.
*   **Внедрение заголовков:** Это ключевое изменение позволило клиентам отправлять метаданные вместе с запросом (например, информацию о типе устройства или требуемом формате ответа).
*   **Новые методы:** К GET добавились методы **POST** и **HEAD**, что позволило не только запрашивать, но и отправлять данные на сервер (например, через формы).
*   **Типы контента:** Запросы теперь могли работать не только с HTML, но и с изображениями, видео и обычным текстом.
*   **Ограничение соединения:** Основным недостатком оставалось то, что для каждого отдельного запроса требовалось открывать новое TCP-соединение, что создавало огромные задержки из-за «трехэтапного рукопожатия».

##### **HTTP/1.1 (1997–1999): Оптимизация и обязательные параметры**
HTTP/1.1 стал стандартом на долгие годы, внеся важные уточнения в структуру сообщения.
*   **Обязательный заголовок Host:** Начиная с этой версии, клиент обязан указывать заголовок `Host` в запросе, что позволило размещать несколько сайтов на одном IP-адресе.
*   **Новые методы управления:** Появились методы **PUT, PATCH, OPTIONS и DELETE**, расширившие возможности управления ресурсами на сервере.
*   **Устойчивые соединения:** В отличие от 1.0, здесь соединения стали постоянными по умолчанию. Это позволило отправлять несколько запросов последовательно через одно и то же открытое соединение.
*   **Конвейерная обработка (Pipelining):** Клиент получил возможность отправлять несколько запросов, не дожидаясь ответа на первый, хотя это часто приводило к проблемам блокировки очереди (Head-of-Line blocking).

##### **HTTP/2 (2015): Бинарная революция**
HTTP/2 радикально изменил способ передачи запроса, уйдя от чисто текстового формата.
*   **Бинарный формат:** Запросы перестали быть читаемыми для человека. Теперь сообщение состоит из **фреймов** (бинарных порций данных) и **стримов** (потоков).
*   **Мультиплексирование:** Это позволило отправлять множество запросов асинхронно через одно TCP-соединение. Ответы также приходят асинхронно, и клиент собирает их по ID потока.
*   **Сжатие заголовков (HPACK):** Поскольку заголовки часто дублируются от запроса к запросу, HTTP/2 сжимает их и использует таблицу индексов, чтобы не пересылать повторяющиеся данные, экономя пропускную способность.
*   **Псевдо-заголовки:** Появились специальные технические заголовки, начинающиеся с двоеточия, такие как `:method`, `:scheme`, `:host` и `:path`, которые заменяют начальную строку запроса из версии 1.x.
*   **Приоритизация:** Клиент теперь может назначить приоритет конкретному потоку (запросу), чтобы сервер понимал, какие данные нужно отдать в первую очередь.

##### **HTTP/3 (2022): Смена транспорта**
HTTP/3 сохраняет логику фреймов и мультиплексирования, но переходит с протокола TCP на **QUIC (на базе UDP)**.
*   **Устранение задержек:** Благодаря QUIC запрос устанавливается быстрее, так как процесс рукопожатия (handshake) совмещен с установкой шифрования.
*   **Миграция соединений:** Запрос в HTTP/3 привязан к **идентификатору соединения**, а не к IP-адресу. Это позволяет, например, продолжать загрузку данных без обрыва, если ваш телефон переключился с Wi-Fi на мобильную сеть.
*   **Решение проблемы блокировки:** В отличие от TCP (в HTTP/2), потеря одного пакета в UDP больше не блокирует все остальные параллельные запросы в этом соединении.

### **Детальный разбор составляющих**

#### **Методы (Verbs)**

Метод определяет характер взаимодействия с сервером:

- **GET:** используется для запроса информации (обычно веб-страницы). Ожидается, что сервер просто вернет данные.
- **POST:** служит для передачи данных на сервер (например, логина/пароля или данных заполненной формы).
- **HEAD:** аналогичен GET, но запрашивает только заголовки ответа без самого тела ресурса.

#### **Заголовки (Headers)**

Заголовки передают служебную информацию в текстовом виде. Они могут содержать данные об авторизации, настройки кэширования, параметры кодировки и информацию о **Cookies** клиента. В HTTP/2 была введена технология **HPACK** для сжатия этих заголовков, чтобы уменьшить избыточность данных.

#### **Тело запроса (Body)**

Этот блок используется, когда клиенту нужно отправить информацию на сервер (например, при методе POST). Тело может содержать данные форм, файлы или JSON-объекты.

### **Ключевые принципы работы**

- **Stateless (Отсутствие состояния):** HTTP является протоколом без сохранения состояния. Это означает, что сервер не хранит информацию о клиенте между транзакциями, и **каждый запрос должен содержать все данные**, необходимые для его выполнения.
- **Соединения:** В ранних версиях каждый запрос требовал открытия нового TCP-соединения, что замедляло работу из-за «трехэтапного рукопожатия». Начиная с HTTP/1.1, стали использоваться **устойчивые соединения (persistent connections)**, позволяющие отправлять несколько запросов последовательно через одно соединение. HTTP/2 и HTTP/3 пошли дальше, используя **мультиплексирование** для асинхронной передачи множества запросов одновременно.
