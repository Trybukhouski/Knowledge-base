> [!abstract] **Главное**
> Результат масштабной эволюции, решивший критические проблемы производительности HTTP/1.1. Главный сдвиг — переход от текстовой передачи данных к **бинарному мультиплексированию**. Теперь множество запросов и ответов передаются асинхронно через одно единственное TCP-соединение, не блокируя друг друга. Протокол стал бинарным (нечитаемым для человека), внедрил эффективное сжатие заголовков (HPACK) и позволил серверу отправлять ресурсы в кэш браузера заранее (Server Push). ^f783aa

## Причины появления HTTP/2: Кризис HTTP/1.1
К 2010-м годам старый стандарт перестал справляться с нагрузками современного интернета:
- **Ресурсоемкость:** Страницы требовали десятки файлов, заставляя браузеры открывать более 30 параллельных соединений.
- **Блокировка очереди (HOL Blocking):** Один медленный запрос в соединении полностью останавливал передачу всех последующих.
- **Избыточность:** Постоянная пересылка одних и тех же текстовых заголовков (Cookie, User-Agent) неэффективно расходовала канал.

Базой для стандарта стала разработка Google — **SPDY (2009)**, сфокусированная на снижении задержек (latency).
## HTTP/2 перевела протокол в бинарный формат и Стримы
В отличие от всех предыдущих текстовых версий, HTTP/2 полностью бинарный.
- **Фреймы и Стримы:** Сообщения разбиваются на бинарные кадры (например, `HEADERS` для метаданных и `DATA` для полезной нагрузки).
- **ID потока (Stream ID):** Каждому «диалогу» присваивается уникальный ID, что позволяет точно собирать пакеты асинхронно.
- **Мультиплексирование:** Данные разных файлов теперь перемешиваются в одном канале. Клиенту не нужно ждать завершения загрузки тяжелого файла, чтобы начать получать другой.
## Как работает оптимизация в HTTP/2
- **Сжатие HPACK:** Клиент и сервер хранят общую таблицу заголовков. Если данные не изменились, передается только короткая ссылка на индекс в таблице. Значения дополнительно сжимаются кодом Хаффмана.
- **Псевдо-заголовки:** Начальная строка запроса заменена техническими заголовками, начинающимися с двоеточия (`:method`, `:path`, `:authority`, `:scheme`).
- **Server Push:** Сервер может принудительно отправить важные ресурсы (стили, скрипты) в кэш браузера через фрейм `PUSH_PROMISE` еще до того, как клиент их запросит.
- **Приоритизация:** Клиент может указать, что, например, CSS важнее картинки, и сервер выделит больше ресурсов на этот поток.
## Безопасность и ограничения в HTTP/2
- **HTTPS как стандарт:** Хотя TLS формально не обязателен в спецификации, браузеры поддерживают HTTP/2 только через защищенное соединение (TLS 1.2+).
- **Проблема на транспортном уровне:** Мультиплексирование решило проблему на уровне приложения, но в TCP при потере одного пакета всё равно останавливаются все потоки. Это ограничение стало стимулом для разработки **HTTP/3** на базе QUIC (UDP).
