Структура диалога в HTTP-протоколе строится на фундаментальной модели **«запрос-ответ» (request-response)**. В этом взаимодействии клиент (обычно веб-браузер) инициирует сессию, отправляя запрос серверу, который затем возвращает сообщение с ответом.

### **1. Структура запроса (Request)**

Запрос — это способ, с помощью которого клиент запрашивает информацию для загрузки ресурса. Типичный HTTP-запрос состоит из следующих элементов:

- **Метод (HTTP Method):** указывает на желаемое действие (например, **GET** для получения данных или **POST** для их отправки).
- **URL:** адрес конкретного ресурса в интернете.
- **Версия HTTP:** указывает версию протокола (например, HTTP/1.1 или HTTP/2).
- **Заголовки запроса (Headers):** содержат текстовую информацию в формате «ключ-значение» о браузере клиента, типах данных и кодировках.
- **Тело сообщения (Body):** необязательный блок, содержащий данные, передаваемые серверу (например, информацию из заполненной формы).

### **2. Структура ответа (Response)**

Ответ — это сообщение, которое клиент получает от сервера в ответ на свой запрос. Оно включает:

- **Код состояния (Status Code):** трехзначное число, указывающее на результат выполнения запроса (например, **200 OK** при успехе или **404 NOT FOUND** при ошибке).
- **Заголовки ответа:** метаданные о передаваемом контенте, такие как формат данных (Content-Type) или язык.
- **Тело ответа:** содержит запрошенные данные (например, HTML-код страницы, изображение или видео).

### **3. Технические особенности взаимодействия**

- **Установление связи:** Прежде чем начнется обмен данными прикладного уровня, происходит **«трехэтапное рукопожатие» (three-way handshake)** через протокол TCP для создания надежного соединения.
- **Отсутствие состояния (Stateless):** Протокол спроектирован так, что каждый запрос автономен и выполняется независимо от предыдущих. Сервер не «запоминает» историю общения с клиентом в рамках самого протокола.
- **Эволюция диалога:**
    - В **HTTP/0.9** диалог был максимально прост: запрос состоял из одной строки, а ответ содержал только HTML без заголовков.
    - В **HTTP/1.1** появились **устойчивые соединения (persistent connections)**, позволяющие отправлять несколько запросов в рамках одного TCP-соединения.
    - В **HTTP/2** диалог перешел на бинарный формат; сообщения разбиваются на **фреймы** (HEADERS для метаданных и DATA для контента), что позволяет обрабатывать несколько запросов асинхронно через одно соединение (**мультиплексирование**).
    - В **HTTP/3** для связи используется протокол **QUIC** на базе UDP, что позволяет сохранять активное соединение даже при смене IP-адреса клиента (например, при переходе с Wi-Fi на мобильную сеть).
