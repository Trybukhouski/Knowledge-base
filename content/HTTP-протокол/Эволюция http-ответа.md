#refactoring #atom 
### HTTP/0.9 (1991)
В этой версии ответ был максимально упрощен. Он мог содержать **только HTML-данные**. Как только передача контента в теле завершалась, сервер немедленно закрывал соединение.
### HTTP/1.0 (1996)
Появилась возможность передавать **различные форматы контента** (не только HTML). Стали использоваться заголовки, описывающие содержимое тела.
### HTTP/1.1 (1997–1999)
Введены механизмы для работы с динамическим контентом:
- **Content-Length:** Заголовок, указывающий точный размер тела в байтах. Это необходимо, чтобы клиент в условиях постоянного соединения понимал, где заканчивается один ответ и начинается другой.
- **Chunked Transfer Encoding (Потоковая передача):** Если сервер не знает итоговый размер динамических данных заранее, он может отправлять тело **по частям (чанками)**. Каждая часть имеет свой размер, а завершается передача пустым чанком с нулевой длиной.
### HTTP/2 (2015)
Тело ответа перестало быть текстовым и стало **бинарным**. Теперь оно передается в виде специальных **DATA-фреймов**. Каждое сообщение разбивается на бинарные куски, имеющие уникальный ID потока (Stream ID), что позволяет передавать части разных тел ответов асинхронно через одно соединение.
### HTTP/3 (2022)
Сохраняет логику фреймов, но передает их внутри пакетов протокола **QUIC** (на базе UDP). В этой версии тело (фреймы данных) находится внутри криптографических «конвертов», обеспечивая встроенную безопасность.
