## Главное про HTTP-запрос
**HTTP-запрос (Request)** — это стандартизированное сообщение, с помощью которого клиент (например, браузер) обращается к серверу за ресурсом или отправляет данные.
Он является частью протокола прикладного уровня HTTP, работающего поверх стека TCP/IP.
## Основные компоненты HTTP-запроса
Типичный HTTP-запрос включает в себя следующие элементы:
- **Метод HTTP (HTTP Method):** определяет действие, которое нужно выполнить над ресурсом
- **URL (Uniform Resource Locator):** адрес запрашиваемого ресурса
- **Версия HTTP:** используемая версия протокола (например, HTTP/1.1, HTTP/2, HTTP/3)
- **Заголовки запроса (Request Headers):** служебные данные в формате «ключ–значение»
- **Тело запроса (Body):** необязательный блок с данными, отправляемыми на сервер

> [!Пример запроса]  
> ```
GET /index.html HTTP/1.1
Host: example.com
User-Agent: Mozilla/5.0 (Windows NT 10.0; Win64; x64)
Accept: text/html,application/xhtml+xml
Accept-Language: ru-RU,ru;q=0.9,en-US;q=0.8,en;q=0.7
Connection: keep-alive
> ```

## Детальный разбор составляющих

### Методы
Метод HTTP определяет тип операции, которую клиент хочет выполнить над ресурсом.
#### Основные методы
##### GET 
Используется для получения данных. Предполагается, что не изменяет состояние сервера.
##### POST 
Используется для отправки данных на сервер (например, формы, JSON, файлы) и может приводить к изменению состояния.
##### HEAD
Аналогичен GET, но возвращает только заголовки ответа без тела. Используется, например, для проверки существования ресурса или получения метаданных.
### Заголовки (Headers)
Заголовки передают служебную информацию о запросе в формате «ключ–значение».
Они могут содержать:
- данные об авторизации
- параметры кэширования
- информацию о типе контента (Content-Type)
- настройки кодировки
- данные о HTTP cookies
### Тело запроса (Body)
Тело запроса используется для передачи данных от клиента к серверу.
Применяется, например:
- при отправке форм
- при работе с API (JSON)
- при загрузке файлов
Чаще всего используется с методами POST, PUT и PATCH.
## Ключевые принципы работы
### Stateless
HTTP является **stateless-протоколом** — он не хранит состояние между запросами.
Это означает, что:
- каждый запрос обрабатывается независимо
- вся необходимая информация должна передаваться в самом запросе
Для сохранения состояния используются дополнительные механизмы, например cookies или токены.
### Persistent connections
В ранних версиях HTTP для каждого запроса устанавливалось новое TCP-соединение, что увеличивало задержки из-за механизма «трёхстороннего рукопожатия».
Начиная с HTTP/1.1 используются **persistent connections** (keep-alive), позволяющие выполнять несколько запросов через одно соединение.
В HTTP/2 и HTTP/3 это развитие продолжилось:
- используется **мультиплексирование**
- несколько запросов передаются параллельно в рамках одного соединения
